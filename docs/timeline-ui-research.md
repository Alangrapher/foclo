# Foclo Time Timeline 呈现方案深度研究

> 研究目标：从「直观性和浏览体验」角度，为 Records 页面设计时间回溯浏览模式  
> 核心命题：「如何让用户在看到界面的瞬间就能理解自己的时间分布」  
> 数据模型：records (id, subject_id, description, start_time, end_time, duration_s) · subjects (id, name, color)  
> 窗口约束：760×620（内容区约 540×580）· 原生 JS/CSS · pywebview · 跨平台

---

## 方案总览矩阵

| # | 方案 | 视觉密度 | 认知负荷 | subject 颜色运用 | 实现复杂度 |
|---|------|---------|---------|----------------|-----------|
| 1 | 水平甘特条带 (Hour Stripe) | ★★★★☆ | ★☆☆☆☆ | 色条直接填充 | 中 |
| 2 | 垂直时间线泳道 (Timeline Swimlane) | ★★★☆☆ | ★★☆☆☆ | 左侧色点 + 行色标 | 中 |
| 3 | 日历热力图 (Calendar Heatmap) | ★★★★★ | ★★★☆☆ | 色阶映射 subject | 高 |
| 4 | 环形日环 (Circular Day Wheel) | ★★☆☆☆ | ★★★☆☆ | 扇形弧色 | 中 |
| 5 | 瀑布摘要卡片 (Waterfall Summary) | ★★★☆☆ | ★☆☆☆☆ | 顶部色条 | 低 |

## 方案一：水平甘特条带（Hour Stripe）

### 核心理念

一天的时间轴从左到右展开为 24 小时的水平条带，每个记录是一条**着色横条**，长度 = duration_s，位置 = start_time。满格 24h 的条带如同一张「时间切片」，用户一眼看到哪个色块多，就明白时间花在哪。

### ASCII Wireframe

```
┌──────────────────────────────────────────────────────┐
│ ◀  2026-07-22  Wed                          ▶ [Day]  │
│                                                      │
│ 00┄┄06┄┄┄┄12┄┄┄┄18┄┄┄┄24                              │
│ ┌────────────────────────────────────────────┐       │
│ │    ██████   ████████████      ████████     │       │
│ │    ██code█  ██design████      ██meet████    │       │
│ │    ██████   ████████████      ████████     │       │
│ └────────────────────────────────────────────┘       │
│   [code] 2.5h   [design] 3h   [meeting] 1.8h         │
│                                                      │
│ ─── 2026-07-21  Tue ────────────────── 6.2h total ───│
│ ┌────────────────────────────────────────────┐       │
│ │ ████████████        ████████████           │       │
│ │ ██research██        ██code███████          │       │
│ │ ████████████        ████████████           │       │
│ └────────────────────────────────────────────┘       │
│                                                      │
│ ─── 2026-07-20  Mon ────────────────── 4.1h total ───│
│ ┌────────────────────────────────────────────┐       │
│ │        ██████████████     ██              │       │
│ │        ██writing██████    ██meeting█       │       │
│ │        ██████████████     ██              │       │
│ └────────────────────────────────────────────┘       │
│                                                      │
│                          [Load more... ▼]             │
└──────────────────────────────────────────────────────┘
```

### 交互方式

- **横向滚动**（非默认）：放大到小时粒度时，可左右拖拽
- **竖向滚动**：按天堆叠，自然滚轮翻页
- **粒度切换**：Day View（24h 条带）↔ Week View（7 天缩略条带并排）↔ Month View（30 天迷你条带表格）
- **hover**：显示记录详情卡片（subject name + description + 精确起止时间）
- **点击**：进入该记录的 inline 编辑（复用当前 editRecord 模式）

### 优点

- **直觉性极强**：横条 = 时间，长度 = 时长，颜色 = 活动类别，零学习成本
- **一眼对比**：不同天之间，色块分布一眼看出「今天 vs 昨天」变化
- **subject 颜色天生承载**：每个色条直接使用 subject.color，无需额外图例
- **空时段即空白**：没记录的时间段就是灰底空白，用户自然看到「这 3 小时去哪了」

### 缺点

- 记录密集时条带可能重叠（需要每个条带单独行 → 高度膨胀）
- 跨夜记录需要特殊处理（跨天标记或中断显示）
- 窄窗口下 24h 刻度标签拥挤

### 实现复杂度：中

- 前端：纯 CSS flex/grid 布局 + canvas 或 div 渲染
- 后端 API：需增加 `get_records_by_date_range(start, end)` 返回完整 ISO 时间
- 关键算法：将 duration_s 映射为水平宽度百分比，处理重叠记录的高度堆叠

### API 需求

- 新增 `get_records_by_date_range(start_date, end_date)` → 返回完整 ISO 格式的 start_time / end_time / duration_s
- 可选：`get_records_aggregated(date, granularity='15min')` → 返回该天 96 格聚合数据（用于缩略视图）

### 默认视图

- **最后有记录的那天**（而非今天——避免空态）
- 粒度：Day View（24h 条带）
- 滚动加载：向上翻看更早日期

## 方案二：垂直时间线泳道（Timeline Swimlane）

### 核心理念

以**纵向时间轴**为主轴（从上到下 00:00 → 23:59），每个 subject 分配一个**泳道列**。记录显示在其 subject 列的对应时间位置。类似 Gantt Chart 但以「天」为单位。

### ASCII Wireframe

```
┌──────────────────────────────────────────────────────┐
│ ◀ 2026-07-22  Wed                           ▶ [Day] │
│                                                      │
│ ┌──────┬──────────┬──────────┬──────────┬──────────┐│
│ │ Time │ Code Rev │  Design  │  Writing │  Meeting  ││
│ ├──────┼──────────┼──────────┼──────────┼──────────┤│
│ │ 08   │          │          │          │          ││
│ │ 09   │ ████████ │          │          │          ││
│ │      │ ██PR#144█│          │          │          ││
│ │ 10   │ ████████ │ ████████ │          │          ││
│ │      │          │ ██mockup█│          │          ││
│ │ 11   │          │ ████████ │          │ ████████ ││
│ │      │          │          │          │ ██sprint█││
│ │ 12   │          │          │          │ ████████ ││
│ │ 13   │          │          │          │          ││
│ │ ...  │          │          │          │          ││
│ └──────┴──────────┴──────────┴──────────┴──────────┘│
│                                                      │
│ total: 6.8h  code:2.5h  design:2h  meeting:1.8h     │
└──────────────────────────────────────────────────────┘
```

### 交互方式

- **纵向滚轮**：上下滚动看一整天
- **泳道横向滚动**：当 subject > 5 时，横向滚动查看所有泳道
- **粒度切换**：Day（上图）↔ Week（7 列缩略泳道）↔ Month（30 天迷你泳道）
- **点击色块**：展开记录详情 / 编辑
- **泳道标题点击**：跳转到该 subject 的汇总

### 优点

- **每个 subject 的时间一目了然**：泳道列天然对比「今天写代码 vs 开会 vs 设计各占多少」
- **重叠时段可视化**：多任务并行时，跨泳道的色块清晰可见
- **时间刻度精确**：纵向刻度自然显示 24 小时，精度可控（30min/15min）

### 缺点

- **subject 数量有限制**（≤5 个泳道可读，>8 个不可用）
- **subject 过多的场景**：需要折叠/分组机制
- **垂直空间利用低**：记录稀疏时大量空白
- **需要更精确的 start_time / end_time 数据**（当前前端只发 duration，后端存完整 ISO）

### 实现复杂度：中

- 前端：CSS grid 布局，每个泳道一列（flex:1），色块用绝对定位或 grid row span
- 后端 API：与方案一相同，需要完整 ISO 时间数据
- 核心算法：将时间范围映射到纵向百分比位置

### API 需求

- `get_records_by_date_range(start, end)` 返回完整 start_time / end_time
- `get_subjects()` 已存在

### 默认视图

- **最后有记录的那天**，Day View
- 水平滚动指示器：当 subject > 4 时，在底部显示浅色滚动条

## 方案三：日历热力图（Calendar Heatmap）

### 核心理念

借鉴 GitHub 贡献图：每个**方格 = 一天**，**颜色深浅 = 当天总记录时长**。多个 subject 则拆分为多个**小方格**或**色块占比**。适合**跨月/跨年**的大局观浏览。

### 交互方式

- **点击方格**：在下方展开该日的详情条带（水平甘特缩略）
- **月份切换**：左右箭头切换月份 ← →
- **年份选择**：下拉选择年份
- **hover**：tooltip 显示「7/22 Wed — 4.2h total」
- **方格细分模式**（进阶）：每个方格切分为 2×2 或 3×3 子格，每个子格代表一个 subject 的时长占比

### 优点

- **最高密度的宏观视图**：一张图看一年，瞬间发现「这个月特别忙 / 那个月荒废了」
- **模式识别**：周末是否在工作？每周几的规律？一眼可见
- **视觉冲击力强**：空白的方格（没记录的日子）非常显眼

### 缺点

- **无法快速分辨 subject 分布**（需要点击展开或 color-by-subject 切换）
- **认知负荷较高**：用户需要理解颜色深浅与时间的映射关系
- **不适合精确时间查询**：只知道「今天有工作」但不知道具体做了什么
- **需额外 UI 做 drill-down**（点击后切换回日视图）

### 实现复杂度：高

- 前端：需要计算每天每个 subject 的聚合时长 + 配色映射
- CSS：grid 布局 7 列，方格内支持多色块分割
- 后端 API：需新增聚合查询

### API 需求

- 新增 `get_daily_aggregates(year, month)` → 返回该月每天每个 subject 的 total_duration_s
- 新增 `get_yearly_overview(year)` → 返回全年每天的总时长（用于热力着色）

### 默认视图

- **当前月份**（如果本月无记录，退到最近有记录的月份）
- 热力颜色映射：使用默认的渐变（从 transparent → subject 颜色）

## 方案四：环形日环（Circular Day Wheel）

### 核心理念

将一天 24 小时映射为一个**圆形时钟**，每个记录是圆上的一段**彩色弧**。用户天然理解「钟表」的视觉语言——看到哪个颜色占的角度大就知道时间花在哪。类似 Apple Watch 的 Activity Rings 但用于时间追踪。

### ASCII Wireframe

```
┌──────────────────────────────────────────────────────┐
│ ◀  2026-07-22  Wed                         ▶ [Day]  │
│                                                      │
│                   12                                 │
│               ▄▄███▄▄                               │
│           ▄███████████▄           code (2.5h)        │
│         ▄███████████████          ██████████         │
│        █████  code  █████         writing (1.2h)     │
│       ████████   ████████         ██████             │
│      █████████████████████        meeting (1.8h)     │
│      █████████████████████        ██████████         │
│       ████████    ███████                            │
│        █████ meeting█████        total: 5.5h         │
│         ▀██████████████▀                             │
│           ▀██████████▀                               │
│               ▀▀▀▀▀              [Week ▼]           │
│                   6                                  │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### 交互方式

- **悬停弧段**：高亮该记录，tooltip 显示起止时间 + description
- **点击弧段**：展开编辑
- **中心点击**：切换显示模式（24h 完整 vs 仅记录时段）
- **粒度切换**：Day Ring ↔ Week Ring（7 个同心圆环）↔ Month Ring（30 条弧段）
- **拖拽旋转**：环可旋转查看不同时段

### 优点

- **极高辨识度**：钟表直觉，用户零学习成本理解「这个任务占了半天」
- **视觉美观**：有装饰感，适合作为 Records 页面的「仪表盘入口」
- **一天内的比例感极好**：弧长 = 时间比例，直观判断「我半天做了什么」

### 缺点

- **信息密度低**：只能看一天，切到周/月视图时同心环可读性骤降
- **不支持精确时间读取**：只知道「下午有工作」但几点到几点需要 hover
- **subject 过多时（>6）**：弧段太细，颜色区分困难
- **记录重叠时无法展示**（一个时段两个 subject → 只能显示一个或弧段叠加）
- **屏幕利用率低**：环形占位大但有效信息少

### 实现复杂度：中

- 前端：SVG 或 Canvas 绘制弧段，需要三角函数算角度
- 需要完整的 start_time / end_time（确定弧段的起始角度和扫角）
- 颜色使用 subject.color

### API 需求

- `get_records_by_date_range(start, end)` 返回完整 ISO 时间
- `get_records_daily_overview(date)` → 当天分段聚合（可选）

### 默认视图

- **今天**的环形视图
- 当今天无记录时，退到**最后有记录的那天**
- 默认 24h 完整环，灰色底 = 无记录时段

## 方案五：瀑布摘要卡片（Waterfall Summary）

### 核心理念

按**日期倒序**分组，每天一张**摘要卡片**。卡片顶部是 subject 分布的水平色条（像进度条），下方列出该天的所有记录。结合了列表的**可浏览性**和色条的**视觉压缩性**。

### ASCII Wireframe

```
┌──────────────────────────────────────────────────────┐
│                        [All Time]     [Search... 🔍] │
│                                                      │
│ ┌──────────────────────────────────────────────────┐ │
│ │ Tue Jul 22  ─────────────────────── 5.5h total  │ │
│ │ ████████████████████████░░░░░░░░░░░░░░░░░░░░░░  │ │
│ │ ██code███design███meeting█writing               │ │
│ │ 2.5h       1.5h     1.0h     0.5h               │ │
│ │                                                  │ │
│ │ 09:15-11:45  Code Review  PR#144     [detail ▸] │ │
│ │ 11:50-12:30  Design       mockup      [detail ▸] │ │
│ │ 13:15-14:05  Meeting      sprint sync  [detail ▸]│ │
│ │ 14:30-15:00  Writing      doc update   [detail ▸]│ │
│ └──────────────────────────────────────────────────┘ │
│                                                      │
│ ┌──────────────────────────────────────────────────┐ │
│ │ Mon Jul 21  ─────────────────────── 3.2h total  │ │
│ │ ████████████████░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░  │ │
│ │ ██research████████code█                         │ │
│ │ 2.0h          1.2h                              │ │
│ │                                                  │ │
│ │ 10:00-11:30  Research     data analysis [detail▸]│ │
│ │ 14:00-14:45  Code Review  fix bug#212  [detail▸] │ │
│ │ 15:00-15:15  Code Review  deploy v1.2  [detail▸] │ │
│ └──────────────────────────────────────────────────┘ │
│                                                      │
│ [Load 7 more days ▼]                                 │
└──────────────────────────────────────────────────────┘
```

### 交互方式

- **竖向滚动**：瀑布流翻看历史
- **点击日期标题**：跳转到该天的详细信息页（如果未来有）
- **点击色条中的色块**：过滤/高亮该 subject 的记录
- **卡片折叠/展开**：可折叠旧日期，节省空间
- **悬停记录行**：显示 action buttons（编辑/删除，复用现有模式）

### 优点

- **信息层级清晰**：日期 → 颜色概览 → 逐条详情，三步递进
- **一眼看一天**：色条长度的比例告诉你「今天花时间最多的任务是什么」
- **空状态自然**：没记录的日子不出现卡片，不会造成认知负担
- **完美兼容现有数据**：不需要额外 API，现有字段足够

### 缺点

- **信息密度中等**：不如热力图能展示一年，也不如甘特条带精确到半小时
- **subject 过多时色条拥挤**（>6 个 subject 时色条段太细）
- **跨周/月对比**需要用户滚动，缺乏整体概览

### 实现复杂度：低

- 前端：纯 DOM 渲染，每张卡片 = div + flexbox
- 色条：flex items 按比例分配 width
- 记录行：与现有 table row 几乎相同
- 后端：现有 `get_records(filter)` 返回的数据已足够

### API 需求

- 现有 `get_records(filter="all")` 已足够，只需按 date 分组
- 可新增 `get_records_grouped_by_date(limit=30, offset=0)` 优化分页性能

### 默认视图

- **最近 7 天**（所有有记录的日期）
- 超过 7 天显示「Load more」按钮
- 如果今天有记录，今天第一张卡片展开；其余默认折叠仅显示摘要色条

## 推荐实施路径

### Step 1：瀑布卡片（本周内可上线）

作为 Records 页面底部的新视口切换，与现有表格视图共存

- 耗时：~5 天
- 风险：中（布局算法需要处理重叠记录）

## 附录：关键设计决策

### 默认视图选择

| 场景 | 建议默认视图 |
|------|------------|
| 首次使用（无数据） | 瀑布卡片 + 空状态引导「开始计时吧」 |
| 日常使用 | 最后有记录的那天的**甘特条带** |
| 查看一个月 | 热力图 + 点击某天展开甘特条带 |
| 查看很久以前 | 瀑布卡片（按日期分组，避免用户迷失） |

### 时间粒度切换模式

统一使用 Segmented Control 风格切换：

```
[Day] [Week] [Month] [Year]
```

- Day：24h 精密视图（甘特条带 / 环形）
- Week：7 天缩略（方案一/二/五）
- Month：30 天迷你（方案三）
- Year：年度热力（方案三）

### 数据空状态处理

有两种空状态需要区分：

1. **今天没有记录** → 显示「今天还没有记录 🐾」+ 提示框「回到 Timer 开始计时」
2. **选中的一天没有记录** → 显示空白甘特条带（灰色背景）+ 底部提示「这一天没有记录」

### 与现有编辑功能的整合

- 在 Timeline 视图中，hover 记录显示 ✎ 🗑 图标（复用现有 editRecord / delRecord）
- 点击时间条/色块跳转到 inline edit 模式
- 添加记录：在顶部保留现有 add row，但改为时间选择器（起止时间滑块）而非 duration input
